# Cryptographic design
## Key management

LeichFS uses three-level key hierarchy to separate responsibilities and limit sensitive material exposure.

| Level | Name | Salt | Generation | Storage |
|---|---|---|---|---|
| 1 | Wrapping Key (KEK) | conf_salt from `.leichfs.conf` | Derived via Argon2id from passphrase + conf_salt at each init or mount  | Ephemeral - zeroed after use |
| 2 | Master Key | - | Generated by getrandom(2) once at init | AES-256-GCM wrapped in `.leichfs.conf` |
| 3 | File Key | file_salt from file header bytes 16-31 | Derived via HKDF-SHA256 ([RFC 5869](https://datatracker.ietf.org/doc/html/rfc5869)) from Master Key on each `open()` | Ephemeral - in `FH::file_key` zeroed on `close()`  |

## Additional Authenticated Data (AAD)

Each chunk of ciphertext is authenticated with a 40-byte AAD, which is costructed as:

```
AAD[40] = magic[8] || be32(version)[4] || be32(chunk_sz)[4] || salt[16] || be64(chunk_idx)[8]
```

The AAD itself is never stored explicitly on disk. It is reconstructed from the header fields and chunk index. So it binds each chunk's ciphertext to:

- File : via `magic` and `salt`
- Format version: via `version` and `chunk_sz`
- Chunk position : via `chunk_idx`

_This is intentional to prevent chunk-level attacks. See [Chunk](#chunk) section below_

# File Format
## Header

```
 8 bytes  magic          ASCII literal "LEICHFSX"
 4 bytes  version        little-endian uint32, currently 1
 4 bytes  chunk_sz       little-endian uint32, plaintext bytes per chunk (65536)
16 bytes  salt           random per-file salt, used as HKDF input
24 bytes  reserved       zero-filled, reserved for future use
 8 bytes  plain_len_be   big-endian uint64, canonical plaintext byte count
```

Total header size: **64 bytes** (enforced by `static_assert`).


## Chunk

Each chunk stores exactly a single unit of plaintext for up to 65536 bytes (64 KiB). Chunks are stored sequentially, immediately after the header with no gaps between them. 

```
12 bytes  nonce          random[8] || be32(chunk_index)
 N bytes  ciphertext     AES-256-GCM output, N = plaintext length (1–65536)
16 bytes  tag            AES-256-GCM authentication tag
```

### Nonce

Nonce is constructed from 8 random bytes returned by `getrandom(2)`, followed by the chunk index encoded as a big-endian 32-bit integer.

## Offset formula

The byte offset of chunk `i` in the backing file is:

```
offset(i) = header + i × ( nonce + plaintext length in chunk + tag) 
          = 64 + i × (12 + 65536 + 16)
          = 64 + i × 65564
```

For a file with plaintext length `N`:

```
num_chunks        = ceil(N / 65536)
last_chunk_plain  = N mod 65536   (or 65536 if N is an exact multiple)
on_disk_size      = 64 + (num_chunks - 1) × 65564 + 12 + last_chunk_plain + 16
```
## Storage overhead


```
overhead = 64 (header) + num_chunks × 28 (nonce + tag per chunk)
```

For a file smaller than one chunk (N ≤ 65536 bytes):

```
on_disk_size = 92 + N     (64 header + 12 nonce + 16 tag + N ciphertext)
overhead     = 92 bytes   (fixed)
```

For large files the per-byte overhead approaches a constant:

```
28 bytes per 65536 bytes plaintext = 0.043% overhead
```

Comparison with gocryptfs:

|  | LeichFS | gocryptfs |
|---|---|---|
|Overhead| 28 | 32 |
|Block size| 65536 | 4096 |
|Per-block Overhead| 0.043% | 0.781% |

## Examples

**0-byte file:**

```
Header    64 bytes   (plain_len_be = 0, no chunks follow)
Total:    64 bytes
```

**7-byte file (e.g. "secret\n"):**

```
Header    64 bytes
Chunk 0   12 + 7 + 16 = 35 bytes
Total:    99 bytes
```

Verified from `stat test/plain/secret`: size = 99 ✓

**15-byte file (e.g. "secreeeeeeeeet\n"):**

```
Header    64 bytes
Chunk 0   12 + 15 + 16 = 43 bytes
Total:    107 bytes
```

Verified from `stat test/plain/secret2`: size = 107 ✓

**65536-byte file (exactly one full chunk):**

```
Header    64 bytes
Chunk 0   12 + 65536 + 16 = 65564 bytes
Total:    65628 bytes
```

**65537-byte file (one full chunk + one byte):**

```
Header    64 bytes
Chunk 0   12 + 65536 + 16 = 65564 bytes
Chunk 1   12 + 1     + 16 = 29    bytes
Total:    65657 bytes
```

**1 MiB file (1048576 bytes = 16 full chunks):**

```
Header     64 bytes
Chunk 0–15 16 × 65564 = 1049024 bytes
Total:     1049088 bytes
Overhead:  512 bytes (0.049%)
```
